# OMNIA Retry Loop Cross-Model v0
## Same pipeline, different generator

Status: active  
Priority: high  
Scope: compare OMNIA behavior across multiple LLM backends while keeping the full intervention pipeline unchanged

---

## 1. Purpose

This document defines the first cross-model comparison phase for OMNIA.

Goal:

test whether the current OMNIA retry-loop pipeline preserves a positive safety effect when the generator model changes but the rest of the system remains fixed.

This is not a leaderboard exercise.

It is a structural portability test.

The question is not:

Which model is best?

The question is:

Does OMNIA still detect and manage fragile outputs when the source model changes?

---

## 2. Core principle

In this phase, only one variable changes:

- generator model

The following must remain fixed:

- gate logic
- thresholds
- retry policy
- adapter path
- dataset
- prompt structure
- output format
- evaluation rules
- audit trail format

This preserves causal readability.

Rule:

```text
change one variable at a time


---

3. Main question

The key question of this phase is:

Are the structural fragility signals used by OMNIA portable across models of similar class?

Equivalent operational question:

If the same task is generated by two different models under the same pipeline, does OMNIA still deliver:

positive Safety Dividend

bounded Retry Waste

low Over-Defensive Cost

stable intervention routing


If yes, OMNIA becomes meaningfully more model-agnostic inside its tested perimeter.


---

4. What this phase is NOT

This phase is not intended to prove:

universal model neutrality

superiority over all providers

complete calibration invariance

semantic equivalence across models

final production readiness


It is only intended to test portability of the current OMNIA runtime logic across more than one generator.


---

5. Experimental rule

The comparison must preserve the same runtime wrapper.

The canonical logic stays:

1. receive input


2. generate candidate_1


3. run surface acceptance


4. run OMNIA gate


5. branch:

pass

low_confidence_flag

retry

escalate



6. if retry:

generate candidate_2

re-run gate

emit final action



7. compute runtime metrics


8. compare against baseline



No intervention logic should change between models.


---

6. Model selection rule

The first cross-model comparison should use models of comparable practical class.

Do not compare:

a tiny local model

a frontier reasoning model

a highly specialized model

a model with radically different output protocol


The first useful comparison is between:

two general-purpose chat models

similar cost class or neighboring cost class

both capable of structured JSON output

both suitable for low-temperature testing


The comparison should remain interpretable.


---

7. Recommended first comparison shape

Recommended structure:

Model A = current validated reference backend

Model B = second general-purpose backend of similar role


Both should run with:

same temperature

same retry limit

same output schema constraint

same sample batch

same acceptance logic

same OMNIA thresholds


This creates a fair portability test.


---

8. Dataset rule

The cross-model comparison should use the same live-backend dataset first.

Recommended starting point:

reuse data/omnia_retry_loop_real_backend_v0_inputs.jsonl


Reason:

preserves continuity

isolates the generator variable

avoids dataset drift

allows direct comparison of runtime economics


Only after that comparison is stable should the batch be expanded.


---

9. Metrics to compare

The comparison must preserve all metrics from the real-backend phase.

Core runtime metrics

Safety Dividend

Retry Waste

Retry Yield

Baseline Harmful Accepts

Gated Harmful Accepts

Net Harmful Acceptance Reduction

Over-Defensive Cost

Final Action Distribution

Net Effect Distribution


Live operational metrics

Request Count

Mean Latency

Retry Latency Overhead

Total Tokens

Estimated Total Cost

Safety Dividend per Retry

Safety Dividend per Unit Cost


These are the metrics that matter.

Do not replace them with vague qualitative commentary.


---

10. What must be compared explicitly

The cross-model audit must answer these exact questions:

A. Baseline dirtiness

Which model produces the dirtier baseline before OMNIA intervention?

B. Retry usefulness

Which model benefits more from a second chance?

C. Escalation necessity

Which model tends to produce more immediately dangerous outputs?

D. Calibration portability

Do the same thresholds still behave sensibly on both models?

E. Economic efficiency

Does OMNIA remain worth the cost on both backends?

This is the real substance of the phase.


---

11. Success criteria

The cross-model phase is successful if most of the following hold:

1. OMNIA keeps Safety Dividend positive on both models


2. Retry Waste stays bounded on both models


3. Over-Defensive Cost remains low on both models


4. The same thresholds remain interpretable on both models


5. The pipeline produces meaningful differences between models without collapsing the intervention logic


6. The audit remains readable enough to compare behavior model by model



This is enough to claim model portability inside the tested perimeter.


---

12. Failure criteria

The cross-model phase fails if any of the following dominates:

OMNIA works only on one model and collapses on the other

Retry Waste explodes on the second model

Thresholds become erratic and non-portable

Stable cases begin failing on the second backend

Structural categories stop mapping coherently across models

Audit complexity becomes too high to distinguish model behavior from noise


A failed comparison is still useful because it reveals whether OMNIA is more model-dependent than currently believed.

The phase should produce at least these artifacts:

data/omnia_retry_loop_cross_model_v0_model_a_results.jsonl

data/omnia_retry_loop_cross_model_v0_model_b_results.jsonl

Optional supporting artifact:

data/omnia_retry_loop_cross_model_v0_summary.json


The result document must compare the two runs directly.


---

14. Audit format rule

The final results document must include:

per-model summary

side-by-side metric comparison

hardest silent-failure cases

stable-pass cases

retry-success cases

worst retry-waste case

threshold portability commentary

cost comparison

latency comparison


This prevents the comparison from degenerating into a shallow "Model A vs Model B" scoreboard.


---

15. Interpretation rule

A difference between models is not automatically a problem.

Expected differences include:

one model may produce cleaner baseline outputs

one model may recover better after retry

one model may generate more compact JSON

one model may over-explain and trigger more fragility signals


The real question is not whether the models are identical.

The real question is whether OMNIA still functions as a coherent intervention layer across both.


---

16. Minimal implementation strategy

The first cross-model implementation should not rewrite the runtime script entirely.

Preferred strategy:

reuse the real-backend script

make model selection configurable by environment variable

run once with Model A

run once with Model B

save outputs to separate result files

compare in a single report


This minimizes unnecessary code drift.


---

17. Non-goals

Do not in this phase:

compare more than two models at once

tune thresholds separately per model

change the prompt between models

expand to a second domain

enlarge the batch aggressively

optimize for benchmark theater


The point is portability, not spectacle.


---

18. Final rule

This phase decides whether OMNIA is merely backend-compatible or genuinely model-portable.

If the gain remains positive across both models under the same pipeline, the project earns the right to say that OMNIA is model-agnostic inside its tested structured-output runtime perimeter.

If not, the limitation must be recorded clearly and treated as structural evidence.
